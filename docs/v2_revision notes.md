> Revision notes (2026-05-25):
> * Pass 1 (initial post-audit, findings 1ŌĆō10):
>   * **F1** (architectural simplification ŌĆö the most consequential change vs. the earlier v2 draft): removed `forecast_hw_seasonal` entirely. The periodic-pattern forecaster is now `prophet` with the hourly regressor, which already captures the seasonal level cleanly without the design ambiguities of the HW seeding scheme (warm-up boundary, what to do when the operator's `historyMinutes` was less than one full season, the conflict between HW's additive/multiplicative seasonality assumption and Prophet's hour-baseline regressor for the same workload). Updated ┬¦1-2 overview/scope, ┬¦3 architecture diagram, ┬¦4 webhook `preferredForecaster` enum, ┬¦5 hot-path response enum and dispatch (deleted the `forecast_hw_seasonal` subsection, safety-cap row, and warm-up clause), ┬¦6.1 ŌåÆ ┬¦7 pattern ŌåÆ forecaster mapping (`periodic` now always picks `prophet`), and the ┬¦9 failure-mode table.
>   * **F2a** (cold-path threshold semantics): the previous v2 draft labeled `CLASSIFIER_HIGH_CONFIDENCE_POINTS=240` as "~4h" and `CLASSIFIER_MIN_POINTS=70` as "~6h before first classification" ŌĆö both numbers were ported from a 1-min-resolution prototype but weren't rescaled when the cold path moved to 5-min downsampling (1440 ŌåÆ 288 points over 24h). Initial fix in this pass: `MIN_POINTS=24` (~2h), `HIGH_CONFIDENCE_POINTS=48` (~4h) so the labels match reality. (User pushed back on this in Pass 2 ŌĆö see F2a-revisited.)
>   * **F3a** (Prophet clock contract): `forecast_prophet` builds a synthetic `ds` array of 1-minute timestamps "ending now", but the spec didn't say whose "now" ŌĆö the controller's, the Forecast Service's `datetime.now()`, or the timestamp of the last `rps_history` point. Pod clock skew between controller and Forecast Service would silently misalign the hourly regressor's lookup. Pinned the contract: `ds` is reconstructed so that `utc_hour_of(ds[-1]) == context.current_hour_utc`. Earlier values step backwards in 1-minute increments. The Forecast Service does not call its own clock. (Pass 3 F17 later closed the minute-level gap left open here.)
>   * **F4a** (autocorrelation lag formula): the ┬¦7 footnote read *"`MIN_POINTS=70` holds for both 1-min and 5-min resolutions"* ŌĆö which became wrong the moment F2a lowered `MIN_POINTS` and inconsistent with the 22-point lower bound the autocorr formula actually needs (lag-12 + 10-overlap minimum). Replaced the footnote with the explicit lag formula in ┬¦7: *"`hourly_autocorr` lag = `60 / CONTEXT_DOWNSAMPLE_RESOLUTION_MIN` points (default: 12). Sets to 0 when `len(series) < lag + 10`."* Decoupled the autocorr gate from `CLASSIFIER_MIN_POINTS` so future tuning of either threshold doesn't silently break the other.
>   * **F5** (Prophet auto-mode knife-edge): `PROPHET_MIN_POINTS=60` against `HOT_PATH_HISTORY_MINUTES=60` meant Prophet only ever engaged exactly when the full hot-path window was filled ŌĆö for the entire warm-up period the controller would silently use `linear_extrap` even when the operator had asked for `prophet`. Lowered `PROPHET_MIN_POINTS` to 30 so Prophet engages halfway through the warm-up; falls back to linear cleanly below.
>   * **F6** (input validation): `current_hour_utc` was specified as `int` 0..23 but had no out-of-range validation in the `/recommend` step-1 input checks. Added explicit *"`current_hour_utc` must be an int in [0, 23]; out-of-range values are treated as malformed context"* ŌĆö folded into the existing "drop context, log a warning" rule rather than rejecting the request.
>   * **F7** (`peakP95RPS` / safety-cap multipliers): considered switching `peakP95RPS` ŌåÆ `peakP90RPS` (lower sensitivity to single-spike outliers) or rebalancing the per-forecaster multipliers (1.5├Ś/2├Ś/2├Ś/3├Ś). Decided **no change** ŌĆö defaults were already reasonable, and the proposed multiplier change would have required re-validating each forecaster's safety behaviour. (No spec edit in this pass; conjugate to F2a ŌĆö once F2a-revisited raised `MIN_POINTS` to 72, the small-sample p95 sensitivity that motivated F7 dissolved.)
>   * **F8** (illusory test path): ┬¦8 had a sentence describing how operators could "test without context" by manually clearing `status.classifiedParams.context`, but no annotation actually guarded the controller from re-populating context on the next reconcile, and the suggested workflow contradicted "ClassifierWorker owns `classifiedParams` exclusively". Either we needed to add a real `autoscaling.agentic.io/skip-context: "true"` annotation (one annotation, one ┬¦4 row, one ┬¦9 row) or delete the misleading sentence. Chose the latter ŌĆö operators can edit the CR status directly for ad-hoc testing, and the value of a dedicated test annotation isn't worth the spec/code surface.
>   * **F9** (forecaster swap at the 12h `hourlyProfileValid` flip): under the previous v2 draft, a periodic workload would silently swap from `forecast_hw_seasonal` to `forecast_prophet` the moment `hourlyProfileValid` flipped from `false` to `true`, with discontinuous prediction behaviour at the boundary. **Auto-resolved by F1**: with `periodic ŌåÆ prophet` regardless of validity, there is no swap. (Recorded here so future readers don't reintroduce the failure mode.)
>   * **F10** (per-request vs. persisted context fields): the `/recommend` context's `current_hour_utc` field was implicitly classed alongside `baselineRPS`, `peakP95RPS`, `trend24hSlope`, and `hourlyProfile`, all of which are persisted in `status.classifiedParams.context`. But `current_hour_utc` is necessarily per-request (it changes every reconcile). Added a one-line clarification in ┬¦5: *"the request body carries `context.current_hour_utc` which the reconciler computes at request time; this field is not persisted in `status.classifiedParams.context`."*
> * Pass 2 (user pushback on F2a + post-review audit, findings 11ŌĆō16):
>   * **F2a-revisited** (the headline change): user objected that `MIN_POINTS=24` from Pass 1 was statistically too thin ŌĆö at 24 5-min buckets, `cv` has ~14% relative error, `peak_to_trough = p99/(mean+1)` is single-outlier-dominated, `hourly_autocorr` has only 12 overlap pairs (correlation barely meaningful), `peakP95RPS` is effectively the second-highest value, and `hourlyProfile` covers ~2 hours so `hourlyProfileValid` is always false. Raised `CLASSIFIER_MIN_POINTS` from 24 ŌåÆ **72** (~6h at 5-min ŌĆö solid recent statistics, more than half a day of context) and `CLASSIFIER_HIGH_CONFIDENCE_POINTS` from 48 ŌåÆ **240** (~20h ŌĆö nearly the full classifier window, ensuring `hourlyProfileValid` is essentially guaranteed at "high"). The `MIN_POINTS >= L + 10` invariant from F4a is still satisfied: 72 Ōēź 12 + 10 = 22 with comfortable margin.
>   * **F11** (`historyPoints: 1440` example): the ┬¦4 status example listed `historyPoints: 1440`, which is `24h ├Ś 60min` at 1-min resolution ŌĆö a v1 leftover that contradicted v2's 5-min downsampling (where the maximum is `24 ├Ś 60 / 5 = 288` points). Fixed to `historyPoints: 264` (~22h at 5-min, near the 288 cap), which is what an operator would actually see on a CR that's been up for ~a day.
>   * **F12** (zero-traffic vs. unclassified conflation): ┬¦6.2's ExplainWorker prompt-conditional read *"Omit the `Long-term context` line when the classifier hasn't run yet (i.e. `BaselineRPS == 0 && PeakP95RPS == 0`)"* ŌĆö but a freshly-deployed workload that genuinely serves zero RPS will have legitimate measurements of `BaselineRPS=0, PeakP95RPS=0`, and we'd suppress the prose line with valid information. Changed the prose-trigger to `Pattern != ""` (the existing convention used elsewhere in the prompt), which is the correct "classifier has run" signal.
>   * **F13** (constant naming ŌĆö spec only): the ┬¦7 cooldown constant `K_TOD_DOWN` is a leftover label from when the feature was called `tod_correlation`; the constant now operates on `hourly_autocorr` and the user-visible pattern label is `periodic`. (The parenthetical was briefly removed during the Pass 6 reviewer edits and restored in the same pass once the source-vs-spec divergence was confirmed to still exist; the spec is intentionally explicit about the divergence rather than silent.)
>   * **F14** (small-sample `peakP95RPS` sensitivity): conjugate to F2a-revisited ŌĆö at low `MIN_POINTS`, p95 over 24-48 buckets is effectively the second-highest value rather than a robust upper-typical, which the per-forecaster safety-cap multipliers (1.5├Ś/2├Ś/2├Ś) implicitly assume. **Auto-resolved by F2a-revisited** raising `MIN_POINTS` to 72; p95 over 72 samples is `value at index ~68` and behaves as a real percentile. No spec edit needed.
>   * **F15** (architecture-diagram drift): the ┬¦3 cold-path box bullet listed *"computes context (baseline, p95, hourly profile)"* but `trend_24h_slope` had been added to context in the v2 draft and was missing from the bullet. Cosmetic but inconsistent with ┬¦4 and ┬¦6.1. Updated to *"computes context (baseline, p95, profile, slope)"*.
>   * **F16** (unused field): the cold path computes `trend_24h_slope`, writes it to `status.classifiedParams.context`, and the hot path forwards it on every `/recommend` call ŌĆö but no forecaster actually consumed it. Two options: drop the field from the schema (YAGNI) or wire it into a forecaster. Chose the latter: blended into `forecast_linear_extrap` as a noise-reduction prior ŌĆö `m_blended = (1 - LINEAR_EXTRAP_TREND_BLEND) * m_window + LINEAR_EXTRAP_TREND_BLEND * trend_24h_slope`. Added the new env var (default 0.30); the bias the blending introduced was caught and fixed in Pass 5 F31 (intercept recomputation) and the unit mismatch was caught in Pass 3 F18.
> * Pass 3 (deeper audit, findings 17ŌĆō24):

> * Pass 5 (final review, findings 31ŌĆō33):
>   * **F31** (bug fix): `forecast_linear_extrap` now recomputes `b = mean(y) - m * mean(x)` after blending `m`. The pass-2 wiring changed `m` without recomputing `b`, which rotated the line around `x=0` instead of around the data's centroid and biased predictions by `(m_blended - m_original) * (n-1)/2` RPS. For a 60-min default window with even modest slope divergence, this was on the order of tens of RPS ŌĆö large enough to materially affect replica decisions.
>   * **F32c** (consistency): `CV_GUARD_MEAN_RPS` moved from ┬¦7 hardcoded constants to ┬¦4 cold-path env vars (operator-tunable; depends on the workload's RPS scale). `GRADUAL_RAMP_DAILY_DRIFT_FRAC` stays as a hardcoded constant in ┬¦7 with a sentence explaining why (it defines what `gradual_ramp` *means* in this autoscaler; tuning it would silently shift the meaning of the pattern label).
>   * **F33** (UX gap from F27b): added prompt conditionals for `max_replicas_binding` and `min_replicas_binding` reasoning tokens. Without these, the LLM saw only the bare `Scaling: 5 ŌåÆ 10 (max_replicas_binding)` line and generated misleading prose ("scaled up to handle load") that hid the actual capacity-planning signal. New lines surface `unboundedRecommended` and the binding CRD bound directly to the operator. Plumbed `UnboundedRecommended`, `MaxReplicas`, `MinReplicas` into ExplainRequest fields.
