# Agentic Autoscaler — Implementation Strategy

**Date:** 2026-05-24
**Status:** Draft for team review
**Companion to:** `docs/design.md` (the system specification)

---

## 1. Purpose and relationship to the design spec

`docs/design.md` is the system spec — what the agentic autoscaler does, how its components interact, and what its parameters are. This document is the implementation strategy — how we build and verify it. It locks in repo layout, language and tooling versions, the test strategy across three tiers of TDD discipline, the eleven implementation plans we will execute (each via the `writing-plans` skill in its own cycle), the merge order and dependency graph between those plans, the synthetic-data approach for tests, the CI workflow shapes, the local development workflow, and the risks we know about up front.

This document does not modify the design spec. If during implementation we discover the design needs to change, we update `docs/design.md` first and link the change from the affected plan.

### Layout notes

- `internal/classifier/` holds pure classification logic; `internal/classifierworker/` holds the goroutine and triggers. The split is a code seam, not a plan seam — both land in plan #5. Same for `internal/explainworker/`.
- `config/` (kubebuilder-generated) and `deploy/` (everything else) are kept separate. Kubebuilder owns its directory; we never edit generated content there by hand.
- Synthetic data is canonical-via-Go-CLI: `hack/synthetic` is the single source of fixtures. Both Go and Python tests read the committed JSON. Cross-language consistency comes from the shared fixture files, not from re-implementing the same RNG in two languages.
- `docs/design.md` moves to `docs/` to keep the repo root tidy. The current `design.md` at the root will move there in plan #1.

## 5. Test strategy

### 5.1 Three tiers of discipline

| Tier | Discipline | Packages and artifacts |
| --- | --- | --- |
| 1 — strict TDD | red → green → refactor at function granularity | `internal/controller/{effective_params,ringbuffer,cooldown,cap}.go`; `internal/classifier/{features,rules,formulae}.go`; `internal/explainworker/{prompt,channel}.go`; `internal/adapters/{prometheus,forecast,ollama}/`; `internal/webhook/validator.go`; `internal/config/`; `forecast-service/src/forecast/{linear_extrap,prophet_model,dispatch}.py`; target-app handlers |
| 2 — test-after, blocking | implement, then write envtest or integration test before merge | `internal/controller/reconciler.go` orchestration; `internal/classifierworker/`; `internal/explainworker/worker.go`; `cmd/controller/main.go` startup |
| 3 — snapshot / lint only | no behavioural tests | kubebuilder-generated deepcopy + RBAC + CRD YAML; `deploy/manifests/*`; `deploy/grafana/agentic-dashboard.json`; Helm values |

### 5.2 Test types and where they run

| Type | Location | Tier | Where it runs | Approx. runtime |
| --- | --- | --- | --- | --- |
| Go unit | `*_test.go` next to source | 1 | every push, locally + PR CI | <2s per package |
| Go adapter contract | inside adapter package, `httptest.Server` stubs | 1 | every push | <2s per adapter |
| Go webhook | `internal/webhook/validator_test.go`, table-driven | 1 | every push | <1s |
| Python unit | `forecast-service/tests/unit/` (pytest + respx) | 1 | every push | <5s |
| Python integration | `forecast-service/tests/integration/` (real prophet on small fixtures) | 1 | every push | ~10s |
| target-app load | `target-app/internal/server/server_test.go`, exercises semaphore→503 path | 1 | every push | <2s |
| Go envtest | `test/envtest/` | 2 | PR CI + locally | 30-60s |
| Snapshot (manifests) | `deploy/`, kubeconform + yamllint | 3 | PR CI | <10s |
| Snapshot (dashboard) | `deploy/grafana/`, JSON schema check + golden-diff | 3 | PR CI | <5s |
| Smoke | `test/smoke/`, kind + apply + healthz + observe one reconcile | 2 | PR CI | ~5 min |
| Quantitative E2E | `test/e2e/`, kind + kube-prom-stack + Ollama + k6 + Prometheus assertions | 2 | nightly + workflow_dispatch | ~25-45 min |

### 5.3 Coverage gates (CI-enforced)

- Tier-1 Go packages: 90% line + 80% branch (golangci-lint with the `gocover` enforcer)
- Tier-1 Python: 90% line (pytest-cov)
- Tier-2 packages: 70% line — uncovered lines must be error-path only (manual check at code review)
- Tier-3: not measured

### 5.4 Universal Definition of Done (every plan must satisfy all of these)

1. All listed components compile (`go build ./...`, `python -m build`, `docker build`).
2. `go vet`, `golangci-lint run`, `ruff check`, `mypy`, `yamllint`, `kubeconform` all pass.
3. Tier-1 packages in scope show red-green-refactor commit history (visible in git log).
4. Tier-1 coverage thresholds met for in-scope packages.
5. Tier-2 envtest covers happy path + at least one error path per new orchestration unit.
6. New env vars added to `README.md` and `docs/runbooks/`.
7. New event reasoning tokens added to `internal/reasoning/` constants and `docs/runbooks/grafana-dashboard.md`.
8. `make smoke` passes locally.
9. PR CI green (lint + unit + envtest + smoke).
10. Plan-specific gates from §7 (1-3 extra checks unique to each plan).

## 6. Synthetic data strategy

A single Go CLI (`hack/synthetic/`) produces every `testdata/*.json` fixture deterministically. Both Go classifier tests and Python forecast-service tests load the same JSON. Cross-language consistency comes from the fixture files, not from re-implementing RNG semantics in two languages.

### 6.1 Generators

```
gen_flat(rng, mean, noise_pct, length)
gen_periodic(rng, base, amplitude, period_minutes, noise_pct, length)
        # period_minutes=60 for the design's `periodic` (hourly) class
gen_spiky(rng, base, burst_freq, burst_mag, noise, length)
gen_gradual_ramp(rng, start, slope_per_min, noise, length)
gen_default(rng, base, noise, length)
```

Each canonical fixture comes in two confidence variants (length=80 → `medium`, length=240 → `high`) plus a `length=50` short-history fixture per pattern that must classify as `pattern_unknown`.

Boundary fixtures live under `testdata/classifier/boundary_*.json`, one per classification rule boundary (`cv≈0.10`, `tod_correlation≈0.70`, `peak_to_trough≈5`, `trend_slope≈2.0`), each with a sibling `_just_under` and `_just_over` pair so off-by-one mistakes in classification rules surface immediately.

### 6.2 Fixture file format (committed under `testdata/`)

```json
{
  "series": [<float>, <float>, ...],
  "label": "flat" | "periodic" | "spiky" | "gradual_ramp" | "default" | "unknown",
  "metadata": {
    "length": 240,
    "seed": 42,
    "generator": "gen_periodic",
    "generator_args": { "period_minutes": 60, "amplitude": 400 }
  }
}
```

The schema is documented in `testdata/SCHEMA.md` and validated by both Go and Python loaders in CI. Any change to the schema is its own PR with both loaders updated atomically.

### 6.3 Regenerating fixtures

```bash
make gen-testdata        # regenerates everything deterministically
go run ./hack/synthetic --pattern=periodic --period-minutes=60 \
    --base=200 --amplitude=400 --noise=0.05 --length=240 \
    --seed=42 --out=testdata/classifier/periodic_hourly.json
```

Regeneration must be idempotent — running `make gen-testdata` twice in a row produces no diff. CI verifies this.

### 6.4 ExplainWorker test fixtures

ExplainWorker prompts depend on `ExplainRequest` shape, not on time series. Hand-authored JSON fixtures live under `testdata/explainworker/` covering each `ReasoningToken` value, `pattern` set vs unset, classifier-confidence variants, and the conditional-line cases (omit `Traffic pattern` when empty/`default`; include "scale was clipped" only on `step_capped_*`).

### 7.2 Plan-specific gates (in addition to the universal DoD)

- **Plan 1.** `kubectl apply -f config/crd/bases/` succeeds; manager pod starts and serves `/healthz`; `kubectl get agenticautoscaler` returns no error; existing `design.md` has been moved to `docs/design.md`.
- **Plan 2.** Every webhook validation rule has a positive and a negative table-driven test case; cert-manager and the webhook are reachable in smoke; rejected `Create` and `Update` requests carry a useful error message.
- **Plan 3.** Each adapter has happy-path + 5xx + timeout + malformed-response tests; Forecast adapter omits `preferred_model` when the value is `"auto"`, `null`, or absent; Prometheus adapter handles both instant and range queries.
- **Plan 4.** envtest covers the cap+cooldown precedence rule from design §5 step 7 ("cooldown wins when both apply"); restart-recovery test seeds in-memory `lastScaleUpTime` / `lastScaleDownTime` from `status.lastScaleTime`; `recommendedReplicas` is published unmodified by cap and cooldown; HPA conflict clears automatically on the next reconcile after the offending HPA is deleted.
- **Plan 5.** Every fixture in `testdata/classifier/` round-trips to its labeled pattern; boundary fixtures (`_just_under` / `_just_over`) classify on the correct side of the boundary; dedup window suppresses redundant generation-change reclassifications even when a classification cycle takes longer than `CLASSIFIER_DEDUP_SECONDS`; manual `reclassify` annotation is auto-removed on success and preserved on `pattern_unknown`.
- **Plan 6.** Every reasoning token in `{scale_up, scale_down, step_capped_up, step_capped_down}` produces a `scale_explained` event when Ollama is reachable; `cooldown_holding_*` does NOT trigger ExplainWorker (per design §5 step 7); Ollama failures (timeout, 404 model-not-found, malformed JSON, empty content) all degrade gracefully with a logged warning and no event; drop-and-replace channel send is verified to discard stale events under sustained burst.
- **Plan 7.** All `testdata/forecaster/*.json` fixtures produce non-negative, finite forecasts; `preferred_model` ∈ {`"auto"`, `null`, absent} all yield identical responses; warm-up reduces first-call latency below `FORECAST_TIMEOUT_SECONDS`; `forecast_prophet_failures_total` counter increments on simulated Prophet failure with linear_extrap fallback path.
- **Plan 8.** Under load above the semaphore limit, the server returns 503 (verified by an N>limit concurrent-driver test); `/metrics` produces well-formed Prometheus exposition; histogram buckets cover 1ms to 10s; `/healthz` responds 200 even under saturation; `/readyz` returns 503 if downstream dependencies are unreachable.
- **Plan 9.** Each scenario passes its own `check()` assertions in `--vus=1` dry-run mode against a httptest server; durations and RPS profiles are configurable via k6 env vars (`RAMP_DURATION`, `RAMP_RPS_PEAK`, `STEADY_RPS`, etc.); both target deployments receive byte-equivalent request streams.
- **Plan 10.** `kubeconform` and `yamllint` clean on all manifests; dashboard imports cleanly into Grafana via the kube-prom-stack sidecar; HPA manifest targets `app-hpa`; AgenticAutoscaler sample CR targets `app-agentic`; both target deployments share an identical PodTemplate spec except for `name`/`labels` (asserted by smoke test diffing the two manifests).
- **Plan 11.** Nightly E2E completes successfully on clean kind under 60 minutes; PR CI completes successfully under 10 minutes; every Makefile target runs cleanly without manual intervention; every runbook has been walked through end-to-end on a fresh box.

## 8. CI workflows

### 8.1 PR + push CI (`.github/workflows/ci.yml`), target under 10 minutes

Triggers: `pull_request`, `push` (to any branch).

Jobs (parallel where independent):

- **lint** — golangci-lint, ruff, mypy, yamllint, kubeconform on rendered manifests.
- **generate-check** — runs `make manifests && make generate` and asserts `git diff --exit-code`. Catches forgotten regenerations.
- **test-go** — Tier-1 Go: unit + adapter contract + webhook tests, with coverage gates from §5.3.
- **test-python** — Tier-1 Python: unit + integration (including small Prophet fits), with coverage gates.
- **test-envtest** — Tier-2 envtest suite for controller, ClassifierWorker, ExplainWorker.
- **build-images** — controller, forecast-service, target-app images via buildx; uses build cache.
- **smoke** — depends on build-images; provisions kind, kind-loads images, applies manifests in dependency order, waits Ready, GETs `/healthz` on every pod, asserts at least one reconcile event observed; ~5 minutes.

### 8.2 Nightly E2E (`.github/workflows/nightly-e2e.yml`), target 45-60 minutes

Triggers: `schedule` (cron `0 2 * * *` UTC), `workflow_dispatch` (manual).

One job. Steps:

1. Provision kind cluster (3 nodes) from `deploy/kind/cluster.yaml`.
2. Helm install kube-prometheus-stack and cert-manager, with the CI Helm values that disable alertmanager and kube-state-metrics to fit the runner budget.
3. Install Ollama on the runner (apt or curl install); cache `~/.ollama/models`; pull `phi3` model.
4. Build and kind-load the three images.
5. Apply CRD, RBAC, webhook, manifests, and the AgenticAutoscaler sample CR.
6. Wait 5 minutes for prom-stack ready + classifier first-run + reconciler steady state.
7. Run k6 ramp scenario for ~25 minutes driving identical RPS to `app-agentic` and `app-hpa`.
8. Query Prometheus over the documented comparison window.
9. Assert: `p99(agentic) <= p99(hpa) * 1.25` AND `5xx_rate(agentic) <= 5xx_rate(hpa) * 1.25`.
10. On failure: collect controller and forecast-service logs, dump Prometheus snapshot, upload as artifacts; post webhook notification if configured.

Tolerances are deliberately loose (`1.25×`) for nightly because of CI variance. A tighter local `make e2e` target uses `1.10×` for runs on a developer's box; `make e2e-strict` uses `1.05×` for release-candidate validation.

## 10. Local development workflow

Documented in `docs/runbooks/kind-bootstrap.md`. Summary:

1. Prereqs: `docker`, `go 1.23`, `python 3.12`, plus what `make tools` installs.
2. One-time setup: `make tools && make ollama-pull` (~2GB model download).
3. Bootstrap: `make kind-up && make install-deps && make images && make kind-load && make deploy`.
4. Drive load: `make k6-ramp`.
5. Iterate on the controller: `make images && make kind-load && kubectl rollout restart deployment/controller -n agentic-system` — full reload in ~30 seconds.
6. Grafana access: `make port-forward-grafana` → `http://localhost:3000`.

Hot-reload caveat: the controller does not hot-reload — every code change requires the rebuild-load-restart cycle. The forecast-service can hot-reload via `uvicorn --reload` outside kind during local dev; the in-cluster deployment uses a fixed image. We accept the rebuild cycle.

Container strategy: `docker build` plus `kind load docker-image`. No registry push. Image tags are `<sha>-dirty` locally and `<sha>` in CI.

## 11. Risks and defensive measures

| # | Risk | Where it lives | Defensive measure |
| --- | --- | --- | --- |
| R1 | Ollama in nightly CI is heavy (~2GB model pull + cold-start cost) | Plan 11 / nightly-e2e.yml | Cache `~/.ollama/models` between runs; nightly uses `phi3` (smaller than llama3.2); local dev keeps `llama3.2` per design |
| R2 | kube-prometheus-stack is ~10 pods + ~5 min settle time; standard runner has 7GB RAM | Plan 10 + 11 | Disable alertmanager + kube-state-metrics in CI Helm values; ephemeral PVC (no `persistence.enabled`); consider larger runner if observed flakes |
| R3 | Prophet's first fit can exceed `FORECAST_TIMEOUT_SECONDS=5s` due to Stan model compilation | Plan 7 | Startup warm-up is mandatory (per design §5); Plan 7 DoD includes an assertion that first real call completes under 5s after warm-up |
| R4 | envtest does not spawn pods, so it cannot validate /scale-actually-happened or restart-during-scale | Plan 4 + smoke | Plan 4 covers the controller-side state machine; smoke and nightly cover the cluster-side reality |
| R5 | cert-manager bootstrap ordering on a fresh kind cluster (own certs depend on own webhook) | Plan 2 + 11 | Smoke runbook applies cert-manager first → waits for `cmctl check api` → applies our Issuer/Certificate → applies our webhook, in strict order |
| R6 | Generation-change-watcher initial-sync race vs `CLASSIFIER_DEDUP_SECONDS=60` if classification takes >60s | Plan 5 | Plan 5 envtest forces a slow Prometheus mock to push classification past the dedup window and asserts no double-classify |
| R7 | Restart-recovery requires atomic seeding of in-memory state from `status` before any scaling decision; bug-prone | Plan 4 | Plan 4 envtest specifically covers "controller restart mid-flight; status survives, in-memory state restored, no spurious cooldown bypass" |
| R8 | app-agentic and app-hpa must be byte-identical except for ownership; any drift skews the comparison | Plan 10 | Plan 10 includes a smoke assertion that the two PodTemplate specs differ only on `name` and `labels` keys |
| R9 | Grafana dashboard import via API is fragile | Plan 10 | Dashboard ships as a ConfigMap with `grafana_dashboard=1` label; auto-loaded by the kube-prom-stack sidecar; `make dashboard-import` is a fallback for non-stack setups |
| R10 | Boundary conditions in classifier rules (`cv=0.10`, `tod_correlation=0.70`, `peak_to_trough=5`, `trend_slope=2.0`) — off-by-one is easy | Plan 5 | Each boundary has a `_just_under`, `_at`, and `_just_over` fixture trio; classifier asserts the right side of every boundary |
| R11 | Cross-language fixture format must be stable across Go and Python loaders | Plan 11 | One JSON schema documented in `testdata/SCHEMA.md`; both Go and Python loaders validated against the schema in CI; any schema change is its own PR with both loaders updated atomically |
| R12 | Forecast-service may OOM under k6 load if Prophet keeps fitting on big histories | Plan 7 + 10 | Set forecast-service Deployment `resources.limits.memory=1Gi`; smoke asserts the pod does not OOMKill during a 5-minute k6 ramp |
| R13 | `phi3` may produce lower-quality prose than `llama3.2`, weakening the qualitative validation of `scale_explained` events | Plan 11 | Nightly only asserts the event was emitted, not its content; local dev uses `llama3.2` and operators can read the actual prose; quality-of-prose is a manual-review gate |
| R14 | Initial-trigger ordering at controller startup: classifier immediate-first-run, reconciler first reconcile, and Deployment-watcher initial-sync emit can fire close together; the dedup design assumes one classification per CR boot | Plan 5 | The three signal sources are documented as a single coalesced cycle in the design; envtest includes a "fast boot, slow Prometheus" scenario asserting exactly one classification |
